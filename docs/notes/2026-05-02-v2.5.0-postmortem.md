# v2.4.0 / v2.5.0 で進めた dogfooding の判断ログ

## 背景

v2.3.0（[Pull Request #28](https://github.com/tomio2480/github-workflows/pull/28)）で reviewdog 指摘ゼロでも summary コメントを upsert する機構を導入した．
ユーザから「中央リポジトリ自身にも同じ Markdown lint を適用したい」 という要望が出たのが契機である．
本ノートは v2.4.0 / v2.5.0 を通じて固まった設計判断と運用パターンを再利用可能な知見として残す．

## 判断 1：scope を 2 PR に分けた

[Pull Request #30](https://github.com/tomio2480/github-workflows/pull/30)（v2.4.0：`post-summary` opt-out）と [Pull Request #31](https://github.com/tomio2480/github-workflows/pull/31)（v2.5.0：`markdown-ignore` + dogfooding workflow）に分けた．

判断軸は次のとおり．

- v2.4.0 は self-test の summary 競合を予防する基盤．caller 形式の md-lint.yml を入れる前段として独立して価値がある
- v2.5.0 は応用層．`markdown-ignore` の設計と dogfooding workflow の追加が同時並行する規模．レビュー観点も多い
- 1 本の PR にすると change 量が膨らみレビューサイクルが長引く．2 段に分けると Red-Green の単位が小さくなりレビュー bot の指摘も焦点が絞れる

教訓として：「複数の独立した変更は scope を割って小刻みに進めるほうがレビューサイクルが短く済む．scope の境界は『単独で価値が生まれるか』 で判定する」．

## 判断 2：multi-line markdown-glob から markdown-ignore 別 input への pivot

v2.5.0 の当初計画では `markdown-glob` を multi-line（改行区切り）に拡張し `#tests/fixtures/**` を混ぜる案だった．
着手して reviewdog/action-markdownlint への配列展開が YAML expression で困難な点に気付き，**`markdown-ignore` 別 input** に変更した．

理由は次のとおり．

- `reviewdog/action-markdownlint` の `markdownlint_flags` は単一文字列の YAML expression で組み立てる．ここに改行区切り配列を流し込めない
- 改行区切り対応は textlint と count-step に対しては shell ループで処理できるが，reviewdog action が阻害要因になる
- inline コメントは `filter-mode: added` で fixture 行に出ないため，**summary 件数だけ filter できれば実用上は十分**
- したがって reviewdog action は触らず count スクリプト側に `--ignore-glob` を足し，新 input `markdown-ignore` から渡す形が最小侵襲となる

教訓として：「プランは investigation 後に必ず再評価する．動かしてから判明する制約は素直に受け入れて設計を pivot する．計画書を書き直してでも実装着手は遅らせない」．

## 判断 3：upsert race を opt-out で吸収する

self-test の `integration-action` job と新 `md-lint.yml` が同 PR で並走する．
両方が同じ marker `<!-- gh-workflows-lint-summary -->` で upsert すると race する．
v2.4.0 で導入した `post-summary` input を self-test 側で `false` に切り「summary を投稿するのは caller-style 側だけ」 とした．

汎用化すると次のヒューリスティックになる．

> **同一 marker を upsert する仕組みを複数経路で動かすときは
> どれか 1 つだけが書き込む構成にし，他は opt-out する．**
> race 回避を「同期」 で解こうとせず「並走しない」 設計に倒す．

## 判断 4：複数 reviewer の判断割れは decision log に残す

`scripts/count-lint-findings.py` の `_path_matches_ignore` で `endswith("/" + prefix)` 分岐の必要性について self-reviewer agent と Gemini の判断が分かれた．

- self-reviewer：「findings は常にファイル path だから dead」 → 削除推奨
- Gemini：「docstring 網羅性と防御性から残すべき」 → 復活推奨

最終判断は **docstring 整合と防御性を優先** して Gemini 案を採用した．self-review で削除した直後に Gemini で復活させる流れになり，PR の commit 履歴に「削除 → 復活」 の往復が残った．

教訓は次のとおり．

> **複数 reviewer 間の判断が割れた場合，採用 / 却下の理由を明示的に decision log に残す．
> 次回同種の指摘が来たときの参照になり，往復を防ぐ．**

## 判断 5：template バージョンコメント bump をリリース運用のチェックリストに組み込む

`templates/.github/workflows/md-lint.yml` の caller workflow サンプルには `uses: ... @<SHA> # v2.x.y` 形式のバージョンコメントがある．
PR #28（v2.3.0）と PR #30（v2.4.0）の両方で更新漏れが Gemini に指摘された．
release ごとに更新する規律はあったが checklist 化されていなかった．

template バージョンコメントは「同 release で発行されるバージョン」 を指すのが整合的．
古いバージョンを残すと利用者が「post-summary input を持たない古い tag」 を copy する事故につながる．
本 PR の `docs/development-notes.md` 「リリース運用」 節に template bump をチェックリスト化する．

## 代替案と棄却理由

1. **multi-line markdown-glob を実装する（当初計画）**

   reviewdog action 側の YAML expression 制約により実現困難．
   仮に reviewdog action を自前 `markdownlint-cli2 --reporter rdjson | reviewdog -f rdjson` に置き換えれば回避できるが，本 PR のスコープを超える．
   `markdown-ignore` 別 input で代替したため棄却．

2. **root の `.markdownlint-cli2.yaml` / `.textlintignore` を caller-friendly に書き換える（fixtures を ignore）**

   self-test の `integration-action` job が fixture を lint 対象にできなくなる．
   self-test の構造変更を伴うためスコープを膨らませる．
   count スクリプト側に filter を入れる方法が最小侵襲のため棄却．

3. **summary を CodeRabbit 風に常時 GitHub Checks API で表示する**

   PR タイムラインに残らないため利用者が件数を辿りづらい．
   v2.3.0 の段階で却下した方針に従う．
